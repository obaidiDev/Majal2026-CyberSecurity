Day 1 · Foundations

Security is three properties.
Everything else is detail.

Before tools, before attacks, before a single command — every system either holds three properties or it doesn't. Today you'll see each one: build the mechanisms that protect it, and the ones that break it.

Set the frame: we are not memorising definitions. Each property gets a live demo they can touch. The whole day hangs off these three words: Confidentiality, Integrity, Availability.

The C · I · A triad

Three properties a system either has or doesn't

Confidentiality

Can only the right people read it?

Broken by: eavesdropping, stolen credentials, a leaked database. The one you can never undo — once a secret is out, it's out.

Integrity

Is the data exactly what it should be?

Broken by: tampering, forged records, a swapped file. The quietest to break — a wrong number can sit undetected for years.

Availability

Can authorised users reach it when needed?

Broken by: DDoS, ransomware, a full disk. The most expensive event most organisations ever face.

Every control you'll ever configure exists to produce one of these. Every attack exists to break one. When you meet a new tool, ask: which of the three?

Don't lecture the definitions — they'll build each one next. Land only the framing: controls produce a property; attacks break one. Availability is a security property, not "IT's problem".

Confidentiality · keeping secrets

The oldest idea in security: scramble it so only a key-holder can read it

Let them play with the wheel. The Caesar cipher is the "hello world" of encryption: the KEY is the shift. The point to land: the method is simple, and the keyspace is tiny.

Confidentiality · why a small key is no key

Break it yourself

They break real "encryption" with their eyes. The lesson isn't "Caesar is bad" — it's WHY: 25 keys. Then the leap: real crypto keeps the method public and the keyspace impossible. Kerckhoffs's principle, without the name.

Confidentiality · one shared key

Symmetric encryption — fast, but there's a catch

Press Play: the message visibly scrambles, travels, Eve copies gibberish, Bob unlocks with the SAME key. Then "show the problem": the key itself must travel, and Eve grabs it. That's the key-distribution problem — the setup for the next slide.

Confidentiality · a public lock + a private key

Asymmetric encryption — the idea that fixed it

Bob publishes a PUBLIC lock anyone can use; only his PRIVATE key opens it, and it never leaves his machine. Eve has the ciphertext and the public lock, but a public lock only locks. No shared secret ever crosses the wire — this is what HTTPS does on connect.

Confidentiality · agreeing on a secret in the open

Diffie–Hellman, as mixing paint

The magic every student should see once: two people agree on a shared secret while an eavesdropper watches every message — because un-mixing is hard. Move the sliders; the shared swatch always matches on both sides. This is what happens the instant every HTTPS page loads.

Integrity · proof that nothing changed

The fingerprint that shatters if you touch one letter

The single best crypto "aha": type, then change one character and watch ~half the fingerprint flip. A hash is a one-way fingerprint. This is the machine behind every integrity check for the rest of the week.

Integrity · did this file get tampered with?

Same fingerprint = same file. Different = don't trust it.

This is the everyday use of a hash: verify a download, a log, an update wasn't swapped. Have them edit one byte and watch VERIFIED flip to TAMPERED. Integrity is about correctness, not secrecy.

Integrity · how your password is stored

Servers verify your password without ever knowing it

Two ideas: (1) the server stores a hash, not the password — it verifies without knowing. (2) salting: same password, different stored hash, so one leak can't crack millions at once. Toggle the salt live.

Availability · reachable when it matters

Nothing is stolen. Nothing is changed. Nobody can get in.

The property people forget is a security property. Drive the attack slider up and watch real users get dropped; then show the two defences recover it. Tie it to ransomware: same property, different weapon.

Putting it together

Which property just broke?

The reflex that outlasts every tool. For each incident, which of C/I/A failed? The last card breaks two — that's defence-in-depth in miniature. Whatever tool they meet all week, this question comes first.
